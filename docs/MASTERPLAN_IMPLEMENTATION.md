# auvide Masterplan Implementation Guide

Status: reconciled implementation plan for all unfinished work in
`docs/MASTERPLAN.md` and `../GUI_MASTERPLAN.md`

Baseline: foundations complete; runtime/bootstrap and protocol foundations
partially implemented; desktop integration and workstation UX pending

Audience: small coding agents completing one bounded work packet at a time

## 1. Purpose

This document translates the product-level master plan into an executable
technical plan. It is intentionally specific about ownership, interfaces,
dependencies, tests, and completion evidence.

The target is:

1. Publish an installable `auvide` CLI on PyPI.
2. Ship desktop installers that do not require the user to install Python,
   `uv`, FFmpeg, or Real-ESRGAN manually.
3. Complete versioned inspect, recipe-schema, plan, preview, progress,
   capabilities, batch, and diagnostic CLI contracts.
4. Build the accessible preview-centered desktop workstation, durable queue,
   history, resumable cancellation, and guided recovery described in
   `../GUI_MASTERPLAN.md`.
5. Add release automation, updates, signing hooks, and truthful user-facing
   distribution material.

This file does not replace `MASTERPLAN.md`. The master plan owns product intent
and release sequencing; `../GUI_MASTERPLAN.md` owns experience requirements; this
file owns implementation mechanics and is the canonical source of packet IDs.

### 1.1 Packet namespaces

Existing packet IDs are retained so completed work and commit history stay
traceable:

- `R0`, `P3*`, `P4*`, `P5*`: packaging/bootstrap/release packets from the
  original implementation plan. In portfolio terms they primarily implement
  workstreams B, D1–D2, R, and part of C1/C6.
- `C*`: new engine/CLI application-contract packets.
- `D*`: new durable desktop application-service packets.
- `X*`: new desktop experience packets.
- `V*`: verification/integration packets that cross workstreams.

Do not rename historical packets after code has landed. Portfolio mapping is
recorded in Section 7; agent prompts always use the packet IDs in this file.

### 1.2 Source-of-truth order

1. Current code and tests determine whether behavior exists.
2. `MASTERPLAN.md` determines scope and release gates.
3. `../GUI_MASTERPLAN.md` determines product behavior.
4. This guide determines ownership and merge mechanics.

If an agent finds a contradiction, it reports it before expanding scope. A
packet may update the plans when the correction is directly required by its
accepted behavior.

## 2. Working Rules For Agents

Every agent must follow these rules:

- Work on exactly one leaf packet unless explicitly assigned integration.
  Parent packets describe features and are never a useful coding assignment.
- Read the assigned packet, its declared contract section, `MASTERPLAN.md`,
  the relevant part of `../GUI_MASTERPLAN.md`, and the nearest `AGENTS.md`.
- Verify dependencies from source/tests before editing. A checked box or status
  note in this document is orientation, not proof.
- Respect `Primary files` as an ownership boundary. Request coordination before
  editing a listed hotspot or another active packet's primary file.
- Prefer adding a module and narrow registration change over growing
  `cli.py`, `lib.rs`, or `main.ts`.
- Contract producer packets land types/fixtures before consumer packets.
  Consumers must not create temporary competing versions of a pending contract.
- Treat `engine/src/auvide/` as the only engine source. Never edit or commit
  `desktop/src-tauri/engine/`; it is generated by `bun run stage-engine`.
- Do not rewrite unrelated user changes. The worktree may already be dirty.
- Keep commits focused and use Conventional Commit subjects.
- Deliver every completed slice as a signed commit under the maintainer's
  configured identity: verify `git config user.name`, `git config user.email`,
  and signing configuration, use `git commit -S`, then verify the signature.
- Push the signed commit to `origin` before starting another slice. Treat a
  signing or push failure as a blocker and report it in the handoff.
- Keep root `VERSION` and `CHANGELOG.md` current for every user-visible change
  or version change. Set versions with `python scripts/sync_version.py --set
  X.Y.Z` and run `python scripts/sync_version.py --check` before committing.
- Add or update tests in the same packet as behavioral code.
- Run the packet's local checks before handing off.
- Record any new URL, checksum, binary, model, license, environment variable,
  capability, or secret in the relevant documentation.
- Never commit downloaded runtimes, model archives, third-party executables,
  credentials, signing keys, rendered media, or local caches.
- Do not declare a packet complete when a command was skipped. Report the skip
  and reason in the handoff.
- Do not opportunistically refactor adjacent systems. Record follow-ups by
  packet ID or propose a new packet.
- Integrators resolve registration, generated types, and cross-module wiring;
  leaf agents must leave a narrow, tested API and avoid broad rewrites.

Each handoff must contain:

```text
Packet:
Baseline/dependencies verified:
Files changed:
Behavior delivered:
Contract/API changed:
Commands run and results:
Tests added or changed:
Risks or follow-up:
Commit:
Push:
```

### 2.1 Leaf-packet sizing

A leaf packet should normally satisfy all of these:

- One observable behavior or one enabling contract.
- One language/runtime where practical.
- One to four primary source files plus tests/fixtures.
- No more than one shared hotspot.
- Independently testable without a later UI.
- Reviewable as one focused commit.

If a packet requires coordinated Python, Rust, and TypeScript changes, split it
into producer, bridge, and consumer leaves plus a final integration packet.

### 2.2 Contract-first sequence

For every cross-layer feature:

```text
schema/types + fixtures
    -> producer implementation + producer tests
    -> Rust bridge/persistence + Rust tests
    -> frontend state reducer + reducer tests
    -> DOM/view integration + accessibility checks
    -> end-to-end story
```

Do not make a DOM packet responsible for designing the engine schema, or an
engine packet responsible for choosing visual layout.

## 4. Intended Architecture

### 4.1 Source and build artifacts

```text
engine/src/auvide/                  canonical Python engine
engine/tests/                       engine tests
desktop/src/                        desktop UI
desktop/src-tauri/src/              Rust orchestration
desktop/src-tauri/resources/        small tracked manifests/notices only
desktop/src-tauri/engine/           generated engine staging; never tracked
scripts/                            repository release/version utilities
.github/workflows/                  CI, package publish, desktop release
```

### 4.2 Desktop data layout

Rust must derive platform locations through Tauri's path resolver. No code may
construct `%LOCALAPPDATA%`, `~/Library`, or `~/.local` manually.

```text
app_data_dir/
  bootstrap-state.json                installed tool/manifest versions
  runtime/
    python/                          uv-managed Python installation
    venv/                            dedicated engine virtual environment
    state.json                      installed runtime/engine manifest versions
  tools/
    ffmpeg/
    realesrgan/
    rife/                            optional until interpolation is selected
  models/
    realesrgan/
    rife/
  jobs/
    <run-id>/
      recipe.json
      cancel.requested
  logs/
cache_dir/
  downloads/
    <artifact>.part
    <artifact>
temp_dir/
  auvide/
    <source-or-run-id>/              engine scratch; resumable
```

The installed application/resource directory is read-only. In particular,
`uv run --project <bundled-engine>` must not attempt to create `.venv` inside
Tauri resources. Runtime, job-control, and scratch state must also stay out of
the checkout and user-selected/OneDrive-synced media directories.

### 4.3 Runtime execution

The desktop runtime is installed once, then renders use the venv's Python
directly:

```text
bundled uv sidecar
  -> uv python install 3.12 into app_data_dir/runtime/python
  -> uv venv app_data_dir/runtime/venv
  -> uv pip install the staged engine and pinned Pillow into that venv
  -> app_data_dir/runtime/venv/<python> -m auvide.cli ...
```

The bundled `uv` binary is a bootstrapper, not a per-render prerequisite. This
keeps render invocation deterministic and makes cancellation target the engine
process rather than an extra `uv` parent process.

The first implementation may download Python and wheels. The later offline
runtime packet can replace those inputs with python-build-standalone and a
vendored wheelhouse without changing the render protocol.

### 4.4 Tool resolution

CLI users retain PATH discovery. Desktop renders inject explicit overrides:

```text
AUVIDE_FFMPEG
AUVIDE_FFPROBE
AUVIDE_REALESRGAN
AUVIDE_REALESRGAN_MODELS
AUVIDE_RIFE
AUVIDE_RIFE_MODELS
AUVIDE_DATA_DIR
```

`engine/src/auvide/tools.py` resolves a valid explicit override first, then
falls back to PATH/cache discovery. Invalid explicit paths must be reported as
invalid, not silently replaced by a PATH result.

### 4.5 Engine-to-app streams

With `--progress-json`, streams have one purpose each:

- stdout: NDJSON protocol events only.
- stderr: human-readable logs and diagnostics only.

Without `--progress-json`, existing CLI output remains human-readable and
backward compatible.

The Rust backend parses stdout into typed events and emits
`render:progress`. It forwards stderr lines as `render:log`. The frontend must
not infer state from human log strings.

### 4.6 Support matrix gate

The intended public baseline from the master plan is:

- CLI: Python 3.9+ with user-managed FFmpeg and Real-ESRGAN.
- Desktop: Windows 10+, macOS 13+, and mainstream desktop Linux.
- Processing: a Vulkan-capable GPU; CPU mode is diagnostic/fallback behavior,
  not the advertised performance path.
- Bundled desktop Python: 3.12.

R0/P3E/P3I must turn "mainstream desktop Linux" and supported CPU
architectures into an explicit release matrix. Do not advertise a platform or
architecture until its runtime, sidecar, FFmpeg, Real-ESRGAN, model, installer,
and smoke test all exist. The README and release workflow must use the same
matrix.

## 5. Versioned Progress Protocol

### 5.1 Command-line contract

Add:

```text
--progress-json
--run-id <opaque-id>
--cancel-file <absolute-path>
```

`--run-id` is optional for CLI users and generated by the engine when omitted.
The desktop always supplies it. `--cancel-file` is optional and is checked at
safe points and while waiting for long-running subprocesses.

### 5.2 Event envelope

Every stdout line in JSON mode is one complete UTF-8 JSON object:

```json
{
  "protocol": "auvide.progress",
  "version": 1,
  "run_id": "opaque-id",
  "type": "stage_started",
  "stage": "extract"
}
```

Required common fields:

| Field | Type | Rule |
|---|---|---|
| `protocol` | string | Always `auvide.progress` |
| `version` | integer | `1` for the first contract |
| `run_id` | string | Stable for the process |
| `type` | string | One of the event types below |

Event types:

| Type | Required payload |
|---|---|
| `plan` | `input`, `output`, `total_frames`, `total_chunks`, `stages` |
| `stage_started` | `stage`, `ordinal`, `stage_count`; optional `chunk` |
| `progress` | `stage`, `current`, `total`, `unit`; optional `chunk` |
| `stage_completed` | `stage`; optional `chunk` |
| `warning` | `code`, `message` |
| `completed` | `output` |
| `cancelled` | `resumable`, `work_dir` |
| `failed` | `code`, `message` |

Stage identifiers are stable machine values:

```text
probe
extract
stabilize_analysis
upscale
interpolate
encode
concat_mux
cleanup
```

Do not invent frame-level precision when a subprocess cannot provide it.
Initially, extract may report only start/completion and processing may report
completed chunks. The `unit` field distinguishes `frames`, `chunks`, and
`stages`. Because the pipeline processes upscale/interpolate/encode inside each
chunk, stage events may repeat with a `chunk` value; consumers must not assume
that every stage identifier occurs only once.

### 5.3 Compatibility rules

- Additive fields are backward compatible within protocol version 1.
- Removing/renaming fields or changing their type requires version 2.
- Unknown event types must be ignored by the desktop after being logged.
- A malformed stdout line in JSON mode is a protocol error, not a human log.
- Process exit code remains authoritative. Expected cancellation exits `130`.
- Tests compare stable fields and must not depend on timestamps or map order.

### 5.4 Rust and TypeScript types

Rust:

```rust
#[derive(Debug, Clone, serde::Deserialize, serde::Serialize)]
#[serde(tag = "type", rename_all = "snake_case")]
enum ProgressEvent {
    Plan { /* fields */ },
    StageStarted { /* fields */ },
    Progress { /* fields */ },
    StageCompleted { /* fields */ },
    Warning { /* fields */ },
    Completed { /* fields */ },
    Cancelled { /* fields */ },
    Failed { /* fields */ },
}
```

TypeScript uses a matching discriminated union in a dedicated module, not
inline `any` or string inspection in `main.ts`.

## 6. Bootstrap And Tool Manifest

### 6.1 Tracked manifest

Create `desktop/src-tauri/resources/bootstrap-manifest.json`. It is reviewed
source, bundled with the app, and versioned independently from application
state.

Suggested shape:

```json
{
  "schema_version": 1,
  "runtime": {
    "python": "3.12",
    "engine": "0.3.0",
    "pillow": "<exact-version>"
  },
  "targets": {
    "<rust-target-triple>": {
      "uv": {
        "url": "<https-url>",
        "sha256": "<64-lowercase-hex>",
        "archive": "zip",
        "executable": "<archive-relative-path>",
        "license": "MIT OR Apache-2.0"
      },
      "ffmpeg": {
        "url": "<https-url>",
        "sha256": "<64-lowercase-hex>",
        "archive": "zip",
        "ffmpeg": "<archive-relative-path>",
        "ffprobe": "<archive-relative-path>",
        "license": "GPL-3.0-or-later"
      },
      "realesrgan": {
        "url": "<https-url>",
        "sha256": "<64-lowercase-hex>",
        "archive": "zip",
        "executable": "<archive-relative-path>",
        "models": ["<required .param/.bin paths>"],
        "license": "<verified-license>"
      }
    }
  }
}
```

The actual manifest must contain real pinned versions, URLs, and SHA256 values
obtained by the assigned agent. Placeholders are forbidden in a mergeable
manifest. RIFE can be a separate optional artifact so users who do not request
interpolation do not download it. Prefer immutable, project-controlled GitHub
Release assets for production bootstrap reliability. Keep the verified
upstream source/version/license in `THIRD_PARTY_NOTICES.md`; mirroring must not
erase provenance or redistribution obligations.

### 6.2 Download safety requirements

- Accept HTTPS URLs only.
- Use an HTTP client with TLS verification enabled and finite connect/read
  timeouts.
- Stream into `<name>.part`; never hold a large archive fully in memory.
- Emit byte progress when `Content-Length` is available.
- Calculate SHA256 while streaming or immediately after download.
- Refuse extraction on a checksum mismatch and delete the `.part` file.
- Extract into a fresh staging directory.
- Reject absolute paths, `..` traversal, device paths, and archive symlinks.
- Apply a reasonable expanded-size/file-count limit.
- Move the verified staging directory into place atomically where supported.
- Write `state.json` only after executable/model smoke checks succeed.
- A retry must clean incomplete staging while preserving verified downloads.
- A manifest version or checksum change invalidates only the affected artifact.

### 6.3 Readiness checks

Readiness is based on checks, not only file existence:

- Runtime Python launches and imports `auvide` and `PIL`.
- `ffmpeg -version` and `ffprobe -version` exit successfully.
- Real-ESRGAN executable launches with its help/version behavior.
- All model pairs required by the UI's models exist.
- If interpolation is selected, RIFE and its selected model exist.
- An optional 10-frame self-test exercises FFmpeg, Real-ESRGAN, and Vulkan.

System-binary mode applies the same checks to PATH results and stores only the
user preference, not copied binaries.

### 9.3 Release smoke checklist

- Install from a clean account without Python, uv, FFmpeg, or Real-ESRGAN.
- Complete setup and restart without reinstalling.
- Render a short SDR clip to SDR and HDR10.
- Verify resolution, audio, pixel format, primaries, and transfer with FFprobe.
- Cancel during a later chunk and resume.
- Exercise a path containing spaces and non-ASCII user/profile characters.
- Exercise low disk space and denied output permission.
- Exercise system-binary mode.
- Check update with no active render, then during an active render.
- Uninstall and confirm user outputs remain; document whether app data remains.

## 10. Security, Licensing, And Privacy Gates

- CSP and Tauri capabilities remain minimal. Any expansion needs a stated use.
- The frontend never receives arbitrary filesystem or shell capability.
- Rust owns downloads, verification, extraction, and process launch.
- All bootstrap inputs are pinned and SHA256 verified.
- Redirects may not downgrade HTTPS or change to an unreviewed scheme.
- Archive extraction must be traversal-safe.
- Recipe/job filenames use opaque run IDs, not untrusted source names.
- Error payloads must not expose tokens, full environment dumps, or URL query
  secrets.
- GPL FFmpeg remains a separately downloaded/aggregated executable. Release
  artifacts include its license/source-offer obligations as verified by the
  release owner.
- `THIRD_PARTY_NOTICES.md` records tool/runtime/model origin, version, license,
  source URL, archive hash, and whether it is bundled or downloaded.
- Analytics/telemetry remain absent unless separately designed and consented.
